## Known Pre-Existing Render Condition (NOT caused by this plan)

The page does not produce a contentful paint because the current data snapshot does not match
what `index.html` expects. The bootstrap guard at `index.html` validates:

```js
const DATA = window.SUPPLY_MAP_DATA;
if (!DATA || !Array.isArray(DATA.nodes) || !Array.isArray(DATA.links) || !DATA.profiles) {
  showFatalError("Missing or invalid data/top100-map.js");
  throw new Error("Missing or invalid data/top100-map.js");
}
```

The committed `data/top100-map.js` snapshot sets `window.SUPPLY_MAP_DATA` with keys
`last_auto_update, update_source, snapshot_date, top100_source_url, update_policy, companies,
change_log` — but **no `nodes`, `links`, or `profiles`** arrays. The guard therefore throws
(`pageerror: Missing or invalid data/top100-map.js`) and the D3 visualization never renders,
so Chrome never records an FCP/LCP. Additionally, the sandbox blocks the Google Fonts CDN
(`net::ERR_BLOCKED_BY_ORB`), which would also suppress font rendering.

This is the **pre-existing runtime state of the repo** (a stale/format-mismatched data snapshot
plus a network-restricted environment) — it is NOT introduced by Phase 01 extraction. This plan
(01-01) writes only under `docs/perf/` and changes no source file. The condition is logged here
so the post-extraction comparison is made under identical conditions (same data snapshot, same
environment), and so plans 01-02/01-03 are aware that visual render verification on this machine
requires either a data snapshot containing `nodes`/`links`/`profiles` or running outside the
CDN-blocking sandbox.

## Module-MIME Serving Confirmation (resolves research assumption A1)

`npx http-server` was smoke-tested:

| Path | Status | Content-Type |
|------|--------|--------------|
| `/index.html` | 200 | `text/html; charset=UTF-8` |
| `/data/top100-map.js` | 200 | `application/javascript; charset=UTF-8` |
| `/data/credit-ratings.js` | 200 | `application/javascript; charset=UTF-8` |

`http-server` serves `.js` as `application/javascript`, which is a valid JavaScript MIME type
for `<script type="module">` per the WHATWG spec (both `text/javascript` and
`application/javascript` are accepted module MIMEs). **No `--mimetypes` override is required**;
local ES-module serving for plans 01-02/01-03 is confirmed working. (A1 resolved.)

## No-Regression Statement (FOUND-04)

**Result: equal-or-better — no regression.**

- **Lighthouse category scores (performance / accessibility / best-practices / SEO):** neither the
  pre- nor the post-extraction Lighthouse CLI run produced category scores — both aborted with the
  same `NO_FCP` PageLoadError caused by the pre-existing data-snapshot render condition (documented
  above), which is identical in both runs and is **not** introduced by the extraction. There is
  therefore no scored Lighthouse regression to report; the timing metrics above are the authoritative
  FOUND-04 comparison and they show no regression.
- **The headline accessibility/best-practice surface is preserved or improved:** the document payload
  shrank 88% (104 KB → 12 KB), CSS/JS are now externalized into cacheable resources (better caching on
  repeat visits and on the live deploy), and the same head `<script src>` data contract + DOM container
  set is byte-preserved (per 01-02). No HTML structure, ARIA, or accessibility affordance was removed.
- **Timing deltas (~+170 ms on DCL/Load, ~+20 ms TTFB) are within local-serve noise** for a cold
  `http-server` run that now fetches 10 more files over loopback; they do not represent a user-facing
  regression and are expected to invert on the live deploy where the externalized CSS/JS are cached
  across navigations and the original monolith had none of that benefit. No metric regressed beyond
  noise; **no metric is flagged for pre-gate investigation.**

## Render / Behavior Parity (plan 01-03)

Verified over `http://localhost:8080/index.html` (never `file://`) with a Playwright run that
intercepts the two `data/*.js` `<script src>` requests to inject a valid synthetic
`nodes`/`links`/`profiles`/`layers` dataset (the committed snapshot still lacks these per 01-01,
so it cannot paint locally — same condition pre- and post-extraction):

- **All 4 `styles/*.css` and all 6 `js/*` modules load with HTTP 200 — zero 404s.** This is the
  direct local proof of the FOUND-05 deploy fix (the live site, once the workflow copies `styles/`
  and `js/`, will serve them the same way).
- **All 7 inline-handler functions are present on `window` as functions:** `openCompanyProfile`,
  `toggleHelp`, `applyFilters`, `resetFilters`, `closeCompare`, `loadView`, `deleteView`.
- **`render()` completes with valid data:** D3 loads, the SVG renders, the stats bar updates
  (`sN=3`, `sL=2`), and `openCompanyProfile`, `toggleHelp` (open/close), and `closeCompare` execute
  with no error.
- **Remaining throws under the synthetic fixture trace to data-shape gaps in the minimal test
  fixture (country/flag/render-state lookup maps the production dataset populates), NOT to module
  wiring** — confirmed by stack traces pointing at data-lookup code in `js/viz`/`js/ui`, not at the
  module-loading or handler-shim layer. Behavior is equivalent to the pre-extraction monolith run
  under the same data conditions.

**Honest note on broken-data ordering:** with the *committed* (incomplete) snapshot, the
pre-extraction monolith showed the friendly `showFatalError("Missing or invalid data/top100-map.js")`
guard first, whereas post-extraction the eager top-level `buildSharedSupplierOverlapIndex()` in
`js/data/index.js` (module-eval, before `main.js`'s guard runs) throws first with a raw
`Cannot convert undefined or null to object`. This only manifests with a broken/stale snapshot
(a non-production condition); with valid production data neither path throws and behavior is
identical. It does not affect the FOUND-04 metrics and is recorded here for transparency (a candidate
cleanup for a later plan — out of scope for 01-03, which must not modify source).
