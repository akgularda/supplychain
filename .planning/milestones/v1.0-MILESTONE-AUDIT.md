## Pillar 1: TRUST (credibility first)

**Verdict: VERIFIED — implementation is substantive and honest**

### TRUST-01 — Provenance badges on every major figure

`js/trust/index.js` exports `provenanceFor(input, ctx)` which derives
`observed | estimated | unknown | derived` tags SOLELY from the existing confidence string
vocabulary in the data (`high*` → observed, `medium*` → estimated). The function explicitly
guards against fabrication with the comment "NEVER fabricates a tag or a source."
`badgeHtml(prov)` returns accessible HTML with `role="img"` and `aria-label`.
Wired into viz tooltips (`js/viz/index.js`), the company card stat bar (market cap gets
`{marketcap:true}` marker → "observed"), and the company card/compare grid
(`js/ui/index.js`).

The `derived` tag is a first-class fourth option — computed analytics (concentration, criticality,
scenario output) are never promoted to "observed" or "estimated". This is the correct honest
treatment of derived aggregates.

### TRUST-02 — Reachable inline source links

Source FK resolution: `provenanceFor` looks up `input.sourceId / input.sf` in a `sourceIndex`
built from `profile.sources[]`. Source links are only emitted when the URL starts with `http`
(XSS guard). 33 dangling FKs + profiles with empty `sf` degrade gracefully to tag-only, no crash.

### TRUST-03 — Confidence score (0-100%) with age decay

`confidenceScore(input, ctx)` in `js/trust/index.js`:
- Base weights: `observed=90, estimated=65, unknown=25`
- Exponential half-life decay: `mult = max(0.5, 0.5^(ageYears/4))`, applied only when a real
  source year is resolvable via `sourceYear()` in `js/data/index.js`
- Unknown floor: 25 (no figure can score below this without a tag+source)
- Future-year guard: `ageYears = max(0, now - year)` prevents negative age
- Tested for bounds [0,100], monotonic decay over a 50-year fuzz, observed>estimated>unknown
  ordering, the unknown floor, and no-year→no-decay behavior.

### TRUST-04 — Methodology view

`<div id="methodologyModal">` in `index.html` (lines 120-173) contains:
- Data sources section: market cap (companiesmarketcap.com), supply-chain data (SEC filings,
  annual reports, vendor disclosures), macro data (IMF DOTS, World Bank, UN Comtrade)
- Confidence weighting formula explained: base weights + exponential half-life
- Known limits section: ~75 dangling source references, estimated relationships, weekly refresh

### TRUST-05 — Freshness indicator tied to pipeline

`updateStatusIndicator()` in `js/ui/index.js` reads `window.SUPPLY_MAP_DATA.meta.generatedAt`
(set by the auto-update script) and computes age in days: ≤7 → "Updated" (green), ≤14 →
"Aging" (amber), >14 → "Outdated" (red). `<span id="lastUpdated">` displays the formatted date.
Falls back to "Unknown" if meta is absent. This is a real pipeline-backed freshness signal.

### TRUST-06 — Tests covering provenance and confidence math

Six dedicated test files with substantive assertions on real data from `data/top100-map.json`.
Tests use `assert.equal()` and `assert.ok()` with bounds checks, not just existence checks.

## Pillar 3: DEPTH (unique intelligence)

**Verdict: VERIFIED — real data, honest methodology, proper provenance on derived analytics**

### DEPTH-01 — Concentration scoring

`companyConcentration(symbol, opts)` in `js/analytics/index.js`:
- Composite score = `round(100 × (0.6 × HHI + 0.4 × sharedFrac))`, clamped [0,100]
- HHI = 1/k under equal-weight assumption (no per-supplier volume field in data)
- sharedFrac = fraction of suppliers depended on by >1 company (breaks the tie)
- Anchor values verified against real data: GILD=36, NVDA=12
- Displayed in the company card with `Derived` provenance badge (never "Observed")

`sectorConcentration(sector, opts)` reports reuse% + effective supplier count
(avoids the raw-HHI pitfall of returning uninterpretable 1-7 values).

### DEPTH-02 — Bottleneck / criticality analytics

`supplierCriticality(opts)` returns suppliers ranked by fan-in count (number of top-100
companies depending on each supplier), descending. Independent of the editorial `bn` flag
(which is a curated company flag with non-contiguous ranks, not a derived structural metric).
Top chokepoint: "credit and risk data inputs" with fan-in 4.
Displayed in `#chokepointsList` panel with highlight-in-graph capability.

### DEPTH-03 — Taiwan semiconductor stress-test

`runScenario(disruption, ctx)` in `js/analytics/index.js`:
- Disables 5 TSMC-related supplier labels (normalized via `normalizeEntityLabel`)
- Single-hop impact only (v1 scope, documented as such)
- Identifies exactly 7 impacted companies: NVDA, AAPL, AVGO, 000660.KS, AMD, AMAT, KLAC
- Total market cap exposed: $11,360,589,871,184 (derived from real node marketcap fields)
- HHI shifts monotonically: 1/5=0.20 → 1/4=0.25 for each impacted firm
- Severity classification: high (≥50% suppliers lost), medium (≥25%), low (<25%)
`scenario.test.mjs` (9 tests) verifies all anchors against the real frozen dataset.

### DEPTH-04 — Derived analytics carry provenance

All concentration, criticality, and scenario outputs use `{derived: true, n: k}` input to
`provenanceFor()` which returns `tag: "derived"` + link to Methodology.
`badgeHtml` maps `derived` to the `Derived` label and `confidence-medium` CSS class.
The Methodology modal explicitly explains each derived metric and its assumptions.

## Honestly Disclosed Gaps / Tech Debt

These items were found during the audit to be **real limitations** that are **honestly disclosed**
in `LAUNCH.md`, `index.html` methodology, or code comments — they are not hidden.

### 1. Equal-weight HHI assumption (TECH DEBT — disclosed)

The dataset has no per-supplier volume/share field (`l.v = 2` constant for all supplier-input
links). The HHI therefore degenerates to `1/k` under the forced equal-weight assumption.
For all companies with the same supplier count `k`, raw HHI is identical.

**Disclosure status:** Fully and explicitly disclosed in three places:
- `js/analytics/index.js` header comment (lines 7-16, with ASSUMPTION label and formula)
- `index.html` `#methodologyModal` (line 150: "no per-supplier volume field ... equal-weight
  assumption, where it reduces exactly to HHI = 1/k")
- The composite score design (adding `sharedFrac` at 0.4 weight) exists precisely to
  differentiate companies that the raw HHI cannot.

**Impact:** A known analytical simplification; does not invalidate the ranking or mislead.
Suitable for v2 if per-supplier volume data becomes available.

### 2. Local Lighthouse Performance score: 58/100 (TECH DEBT — disclosed)

The score is dominated by FCP/LCP/SI in the 4-5s range on a cold local loopback fetch
(`http://localhost:8080`). TBT is 10ms and CLS is 0 — the page is not janky.

**Disclosure status:** Fully disclosed in `LAUNCH.md` gate section and
`docs/perf/launch-lighthouse-2026-06-21.md` with causal explanation (cold loopback, not CDN;
paint-latency-bound on initial D3 + data load, not interaction performance). The gate policy
explicitly does not block launch on the numeric score.

**Impact:** Expected to improve on the live GitHub Pages CDN-backed deploy. Not a fabrication
or a hidden problem.

### 3. Auto-update pipeline echo quoting bug (KNOWN BUG — disclosed)

`auto-update-data.yml` line 34:
```
echo "timestamp=$(date +%Y%m%d-%H%M%S) >> $GITHUB_OUTPUT"
```
The closing `"` before `>>` is missing, so the `timestamp` output is not captured by
the `steps.timestamp.outputs.timestamp` reference.

**Disclosure status:** Explicitly flagged in `LAUNCH.md` (gate section) with the exact
line number, a correct diagnosis ("mangles that one step's timestamp output"), and a clear
non-blocking assessment ("does not block the weekly refresh, the data validation, the
deploy, or the launch gate").

**Impact:** The cosmetic `timestamp` output is unused downstream in the workflow; the actual
data refresh, validation tests, and commit all work. LAUNCH.md's transparency record is
exemplary here.

### 4. Single-hop scenario (scope boundary — disclosed)

`runScenario()` computes only direct (single-hop) downstream impact. Multi-hop cascade
(second-tier supplier disruption) is out of scope for v1.

**Disclosure status:** Code comment in `js/analytics/index.js` line 241-243 states
"SINGLE-HOP ONLY" and "Multi-hop cascade is deferred (out of scope for v1)". The
Methodology modal's scenario section explains that output is "a structural what-if over
the graph, not a forecast."

**Impact:** Results are correctly framed as structural exposure, not economic loss estimates.

### 5. ~75 dangling source FKs (data quality — disclosed)

About 75 nodes/links reference source IDs that are not listed in the profile's `sources[]`
array. These resolve to `tag-only` (no source link) and score at the Unknown floor (25).

**Disclosure status:** `index.html` `#methodologyModal` Known Limits section, line 168:
"About 75 dangling source references (source FKs that do not resolve to a listed source)
remain in the dataset; figures behind them score at the Unknown floor."

**Impact:** Conservative, not deceptive. Correctly shown as Unknown rather than
inflating confidence.
